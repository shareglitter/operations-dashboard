# CLAUDE.md — Glitter Ops Dashboard

Project context for Claude Code (and any human) picking this up. For end-user /
deployment instructions, see [README.md](README.md); this file focuses on **how it
works, the Airtable API footprint, and what's next**.

## What this is

A single-page operations dashboard for Glitter's president. Cleaning volume,
revenue, block growth/churn, block health, president's KPIs, and financials —
all in one static page. It reads from **Airtable** (the only data API) and is
served as a static site on **Netlify**, typically embedded in a **Softr iframe**.

Audience: **one viewer (the president), roughly weekly.** Every design tradeoff
below is calibrated to that — low traffic, correctness and low-maintenance over
scale.

## Architecture

- **`index.html`** — the whole app. Vanilla JS + Chart.js (loaded from CDN). No
  build step. Historical data is baked in via the `data.js` / `health_data.js` /
  `financials_data.js` script tags; the **current month** is fetched live from
  Airtable on load.
- **`data.js`, `health_data.js`** — auto-generated monthly caches of completed-month
  data. Do not hand-edit; they're regenerated by `refresh.js`.
- **`financials_data.js`** — financial actuals / pro-forma assumptions (hand-maintained).
- **`refresh.js`** — Node script that pulls all history from Airtable and rebuilds
  `data.js` + `health_data.js`. Run monthly by GitHub Actions.
- **`netlify/functions/at-proxy.js`** — serverless proxy. Holds the Airtable token
  server-side and paginates Airtable for the browser. The client never sees the token.
- **`.github/workflows/monthly-refresh.yml`** — GitHub Actions workflow (2nd of each
  month, 10:00 UTC) that runs `refresh.js` and commits the result. Requires the
  `AIRTABLE_TOKEN` GitHub secret to be set, or the job fails.

## Airtable API footprint

Two independent paths hit Airtable. Airtable's binding limit is **5 requests/second
per base** (breach → 429 + 30s lockout); there is no monthly request-count cap on
paid plans, so burst rate is the only real constraint.

**1. Monthly refresh (`refresh.js`, server-side, ~once/month):** three full-table
pulls, paginated at 100/page — Cleaning Log (~486 requests) + Blocks ×2 (~12–20).
~500 sequential requests in a couple of minutes, off-hours.

**2. Live dashboard (`index.html`, per page load):** on load, ~14–18 requests
(current-month cleans + all blocks). First visit to the Metrics tab adds ~8 (and
hits the OpsHub base), Financials adds ~6. All lazy and guarded by `metricsLoaded` /
`financialsLoaded` so they fire at most once per session. **No polling / no
setInterval.** Failures degrade gracefully to the cached historical data.

At one viewer/week this is trivially within limits.

## Work done (2026-07)

Hardening pass focused on the API footprint:

1. **Client-side response cache** (`index.html`, `atProxy`) — 15-min localStorage
   TTL keyed by request payload. A reload/reopen within the window reuses results
   instead of re-pulling. Only successful responses are cached; failures throw
   before caching, so a transient error is never pinned for the TTL.
2. **429 / 5xx retry with backoff** — added to both `refresh.js` (`fetchAirtable`,
   plus a ~220ms inter-page throttle to stay under 5 req/s on the ~486-page cleaning
   pull) and the live proxy (`at-proxy.js`, `fetchAirtable`). Honors `Retry-After`.
   Real 4xx (e.g. 422 bad field name) are **not** retried — they pass straight to the
   existing error path that surfaces Airtable's actual message.

## Next steps / deferred

**Deferred: single-request current-month summary (rollup).** `fetchCurrentMonthCleans`
([index.html](index.html)) pulls ~836 current-month rows just to compute 8 aggregates
split core/project/grant (counts, revenue, COGS, trash/debris bags via the categorical
`BAG_MAP`). Airtable's REST API has no GROUP BY, so collapsing this to **1 request**
requires a **month-keyed summary table** with rollups:

- On **Cleaning Log**, add formula helper fields turning categoricals into numbers:
  `Is Core / Is Project / Is Grant` (from `Project`), `Rev` (`50*Multiplier`),
  `COGS` (`IF({Payout},{Payout},25)`), `Trash Bags` / `Debris Bags`
  (`SWITCH` mirroring `BAG_MAP`), and `Month Key` (`DATETIME_FORMAT(...,"YYYY-MM")`).
- Create a **Monthly Cleaning Summary** table, one row per month, linked from
  Cleaning Log via a find-or-create automation on `Month Key`. Add `SUM(values)`
  **rollups** for each helper field.
- Then `fetchCurrentMonthCleans` becomes a single filtered fetch of one summary row.

**Verdict: not worth it at current usage** (one viewer/week; saves ~7 requests/week
and adds a fragile automation). Build it only if the dashboard opens to more viewers
or lands on a high-traffic embed. Full field-by-field recipe is in the chat history
that produced this file.

**Other loose ends:**
- Verify the `AIRTABLE_TOKEN` GitHub secret is set — the monthly workflow (now at
  `.github/workflows/`) needs it. Before the workflow was moved there it never ran,
  so `data.js` history may only be as fresh as the last manual `node refresh.js`.
- `financials_data.js` is hand-maintained — update it when new actuals land.
- `index copy.html` is a stale backup; delete once you're confident in `index.html`.

## Conventions

- No build step, no framework, no npm install for the app itself — keep it that way.
- The Airtable token must never appear in any committed file or client-side code.
- All Airtable access from the browser goes through `at-proxy.js`, never direct.
- `data.js` / `health_data.js` are generated — regenerate via `refresh.js`, don't hand-edit.
